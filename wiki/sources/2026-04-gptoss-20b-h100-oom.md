Identifies a real OOM in vLLM 0.18.0's KV-cache sizer on 80 GB H100 SXM with gpt-oss-20B. The model + KV fit fine; the failure is in the *post-KV* sampler-warmup step that allocates ~786 MiB the sizer didn't budget for. **Fix shipped: GPU_MEM=0.85 default in `vllm-tune` baseline + optimized + cli paths.** The bug is hidden on 94 GB H100 NVL and 141 GB H200 (sufficient slack); only 80 GB SXM trips it. vLLM 0.19 fixes the underlying budgeting (default `VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS=1`).

## Overview

The doc explains why every replica dies at the same line ~30-50 s into boot on 80 GB H100 SXM:
- Model weights (13.6 GiB MXFP4 + Triton dequant workspace)
- KV cache (52.7 GiB at 0.88 budget)
- CUDA graph private pools (0.61 GiB)
- PyTorch reserved-but-unused fragmentation (1.72 GiB)
- "Other" — CUDA context, kernels, FlashAttention scratch (~10 GiB)
sum to 78.7 GiB. After all that, vLLM warms up the sampler with a fake forward pass at `max_num_seqs=1024`, where this line OOMs:

```python
# vllm/v1/sample/sampler.py:90
logits = logits.to(torch.float32)
```

For gpt-oss vocab ~201k, at `max_num_seqs=1024`, this fp32 cast = 1024 × 201,088 × 4 bytes ≈ **786 MiB** — exactly the OOM amount. The bf16 logits already exist (~393 MiB); the cast doubles them to fp32, killing the process.

## Key claims

1. **vLLM 0.18's KV-cache sizer (`vllm/v1/engine/core.py::_initialize_kv_caches`)** runs a memory profile *before* CUDA graph capture and *before* sampler warmup. It hands `(gpu_total × gpu_memory_utilization) − non_kv_peak` to KV cache. CUDA graph private pools and the sampler-warmup workspace land *after* this measurement, pushing the process past the nominal limit.
2. **vLLM itself logs the upstream fix during boot:** `In v0.19, CUDA graph memory profiling will be enabled by default (VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS=1)... To try it now, set ... and increase --gpu-memory-utilization from 0.8800 to 0.9637 to maintain the same effective KV cache size.`
3. **The OOM is hardware-tier-dependent.** Slack table: H100 SXM 80 GiB ~0.4 GiB free → OOM; H100 NVL 94 GiB ~15 GiB free → fine; H200 141 GiB ~62 GiB free → fine. The sampler test fits whenever there's > ~1 GiB of slack.
4. **Reproducibility:** 5/5 OOM at GPU_MEM=0.9 default, 1/1 OOM at 0.88, all on driver 535. At 0.85 weight load + CUDA graph capture complete, sampler warmup passes, requests serve.
5. **`max_num_seqs=256` was rejected as the alternative fix** because it changes the engine's advertised concurrency ceiling — more surprising to a customer than a smaller KV cache.
6. **vLLM 0.19's MXFP4 MoE Triton-backend regression on 120B was the reason for staying on 0.18** at the time of this doc. (The [v019 findings](2026-04-gptoss-20b-v019-findings.md) later showed the regression does not bite 20B, freeing the move to 0.19 + GPU_MEM=0.95.)

## Key data points

### Process memory at OOM (GPU_MEM=0.88, single replica, identical at 0.9)

| Bucket | Size | Source |
|---|---:|---|
| Model weights | 13.64 GiB | MXFP4 20B + Triton dequant workspace |
| KV cache (allocated) | 52.72 GiB | Sized to fill the 0.88 budget |
| CUDA graph private pools | 0.61 GiB | Captured graphs for piecewise decode |
| PyTorch reserved-but-unused | 1.72 GiB | Allocator fragmentation overhead |
| Other (CUDA context, kernels) | ~10.0 GiB | Activation buffers, FlashAttention scratch |
| **Process total** | **78.72 GiB** | (~0.4 GiB free) |
| **Sampler warmup needs** | **+0.79 GiB** | logits.to(float32) at max_num_seqs=1024 |

### Tier-dependent slack table

| Card | Total | 0.9 budget | After model + KV | Headroom for sampler/CUDA-graphs |
|---|---:|---:|---:|---:|
| H100 SXM | 80 GiB | ~71 GiB | ~78.7 GiB used | ~0.4 GiB → **OOM** |
| H100 NVL | 94 GiB | ~85 GiB | ~78.7 GiB used | ~15 GiB → fine |
| H200 | 141 GiB | ~127 GiB | ~78.7 GiB used | ~62 GiB → fine |

### Defaults shipped after fix

| Path | Old default | New default |
|---|---|---|
| `deploy/vllm-baseline/entrypoint.sh` | (vLLM default 0.9 — broken) | 0.85 |
| `deploy/vllm-optimized/entrypoint.sh` | 0.95 (also OOMs) | 0.85 |
| `deploy/vllmtune/src/vllmtune/cli.py` | 0.95 | 0.85 |

## Techniques referenced

- vLLM v1 KV-cache sizer (`_initialize_kv_caches`)
- vLLM CUDA-graph budgeting (`VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS`)
- Sampler dummy-forward warmup at `max_num_seqs`
- bf16 → fp32 logit cast (per-step in vLLM v1 sampler)
- Paged KV cache allocation (post-profile)
- MXFP4 weight loading + Triton dequant scratch

## Gaps & caveats

1. **Driver 535 detail:** the doc reproduces the OOM on driver 535. On bare pip-vllm, an unrelated torch `_inductor.standalone_compile.FakeTensorMode` shadow bug also fires; the official `vllm/vllm-openai:v0.18.0` Docker image avoids it by pinning a working torch wheel.
2. **The 4 GiB slack at GPU_MEM=0.85 is sized for typical workloads.** Customers tuning back up to 0.88 may work on some hardware but reproducibly OOM on 80 GiB SXM.
3. **0.95 will always OOM on 80 GiB cards with this model.**
4. **94 GiB NVL and 141 GiB H200 customers can safely raise GPU_MEM** for more KV headroom — the bug doesn't bite them.
5. The fix is a workaround, not a root-cause patch. The underlying sizer-misaccounting bug is fixed only in vLLM 0.19.
