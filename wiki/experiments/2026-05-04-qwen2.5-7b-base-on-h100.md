First non-gpt-oss target through the Mac-driver loop. Tested whether the autopilot generalizes beyond the gpt-oss-20B 9-config matrix when `bench/scripts/.../run_matrix.sh` doesn't apply. Two of three workloads ran cleanly; the third (`prefill @ c=1024`) failed with a **client-side ulimit `Errno 24: Too many open files`** — not a server problem. Verdict is `inconclusive` rather than `refuted` because the prefill cell wasn't a real test of the BASE configuration.

## Hypothesis under test

[`qwen2.5-7b-base-on-h100`](../hypotheses/qwen2.5-7b-base-on-h100.md): stock vLLM 0.19.0 (`--gpu-memory-utilization 0.9`, no other flags) serves `Qwen/Qwen2.5-7B-Instruct` on a single H100 80 GB SXM and completes a per-replica concurrency sweep across decode @ c=256, prefill @ c=1024, sharegpt @ c=512 with 100% request success. Pass condition: all three cells return `success_pct=100.0` with non-zero `output_tok_s`.

## Setup

**Host:** `h100-1` (`adarsh@89.169.110.238`) — 1× NVIDIA H100 80 GB HBM3, driver 580.126.09, CUDA 13.0.

**Image:** `vllm/vllm-openai:v0.19.0` (already present on box).

**Model on disk:** `/srv/models/Qwen/Qwen2.5-7B-Instruct/` (~15 GB BF16, 4 safetensors shards).

**Container launch (verbatim):**

```bash
sudo docker run -d --name qwen7b-bench --gpus all --ipc=host --shm-size 16g \
    -p 8000:8000 -v /srv/models:/models:ro \
    --entrypoint python3 vllm/vllm-openai:v0.19.0 \
      -m vllm.entrypoints.openai.api_server \
      --model /models/Qwen/Qwen2.5-7B-Instruct \
      --served-model-name qwen \
      --host 0.0.0.0 --port 8000 \
      --tensor-parallel-size 1 \
      --trust-remote-code \
      --gpu-memory-utilization 0.9
```

Cold start: model loaded, `/health` ready in ~2 minutes. Peak HBM in use: 74.3 GB (weights + KV cache + CUDA graphs).

**Sweep loop (verbatim, executed inside the container's host):**

```bash
SWEEP="bench/scripts/benchmark/sweep_api_providers_evalscope.py"
TOK="/srv/models/Qwen/Qwen2.5-7B-Instruct"
SHAREGPT="$HOME/vllm-tune/data/datasets/sharegpt_prompts.txt"
for pair in "decode 256" "prefill 1024" "sharegpt 512"; do
  W=$(echo $pair | cut -d" " -f1); C=$(echo $pair | cut -d" " -f2)
  python3 "$SWEEP" --self-urls http://localhost:8000 --providers self \
      --self-model qwen --workloads "$W" --concurrency "$C" --n-mode conc \
      --tokenizer-path "$TOK" --sharegpt-path "$SHAREGPT" --cooldown 15 \
      --outputs-root "$OUT/evalscope/BASE" --out "$OUT/per_config/BASE.csv"
done
```

**Diff vs the gpt-oss-20B [round 7 OPT experiment](2026-05-04-gptoss20b-h100-opt.md):** different model, no Eagle3 spec config (no public Qwen-7B Eagle3 draft), `gpu_memory_utilization=0.9` (vs 0.85 for gpt-oss-20B which needed the dropdown to dodge a v0.18 OOM bug), per-replica concurrencies bumped 2× because Qwen-7B's smaller weights leave ~62 GB free for KV cache vs gpt-oss-20B's ~40 GB. **Did not use** `bench/scripts/.../run_matrix.sh` — that orchestrator is hardcoded to gpt-oss-20B and isn't usable for a different model.

## Baseline comparison

No prior wiki experiment exists for `Qwen/Qwen2.5-7B-Instruct` on this hardware. This experiment establishes the per-replica datum (or fails to, in the prefill case). Reference points: gpt-oss-20B OPT on the same H100 box hit 13,945 / 3,501 / 16,192 output tok/s on decode/prefill/sharegpt at c=128/512/256 (round 7).

| decode (256 in / 4096 out) | 256 | **7,293** | 1.78 | 1.25 | 33.9 | 140.0 | 143.6 | 100% |
| prefill (4096 in / 512 out) | 1024 | — | — | — | — | — | — | **0% (client error)** |
| sharegpt (~142 in / ~576 out) | 512 | **8,279** | 14.36 | 1.30 | 17.8 | 10.5 | 28.1 | 100% |

**Side-by-side vs gpt-oss-20B OPT** (different model, different concurrency — directional only):

| Workload | gpt-oss-20B OPT @ c=128/512/256 | Qwen-7B BASE @ c=256/—/512 |
|---|---:|---:|
| decode tok/s | 13,945 | 7,293 |

Qwen-7B BASE delivers roughly half the throughput per-replica vs gpt-oss-20B OPT. Expected — gpt-oss-20B OPT had Eagle3 k=3 speculation pulling decode forward; BASE has none. Speculation accounts for the ~2× gap, not raw model speed.

### Failure mode (prefill @ c=1024)

[`evalscope/BASE/self/prefill/c1024/qwen/benchmark.log`](../../raw/benchmarks/2026-05-04-qwen2.5-7b-base-on-h100/evalscope/BASE/self/prefill/c1024/qwen/benchmark.log) shows the cell died on the **client side** with:

```
OSError: [Errno 24] Too many open files
aiohttp.client_exceptions.ClientConnectorError: Cannot connect to host localhost:8000 ssl:default [Too many open files]
sqlite3.OperationalError: unable to open database file
```

Evalscope opens one TCP connection per concurrent request. At c=1024 it tried to open 1024 simultaneous sockets and hit the box's default `ulimit -n` (almost certainly 1024). The vLLM container itself was healthy and served whatever traffic reached it — see boot_logs/BASE.log showing 200 OK entries throughout the period. **This is a client-side benchmark harness ceiling, not a server limitation.**

The metrics dir is empty — no `/metrics` snapshot was captured. (Bug in the bash loop the autopilot wrote — should have been added.)

## Observations

1. **Client ulimit is a hard ceiling at c=1024** for the current evalscope harness. Single-replica benchmarks above c≈900 need `ulimit -n 65536` (or systemd `LimitNOFILE`) on the box before launching the sweep. **File this as `wiki/observations/evalscope-ulimit-ceiling.md`** — it'll bite again on any high-concurrency cell.

2. **`gpu_memory_utilization=0.9` is fine on Qwen-7B + H100 80 GB** under vLLM 0.19. Peak HBM was 74.3 GB; no OOM, no graph-capture issue. Confirms our prediction in the hypothesis that the gpt-oss-20B-specific 0.85 dropdown isn't needed here.

3. **Decode tok/s is heavily TTFT-dominated** at c=256 — TTFT p50 of 1.25 s on a request whose total E2E is 140 s? That's because each request generates 4096 output tokens with `ignore_eos=true`, so it spends 99% of its time in decode loop. The first-token wait is comparatively tiny; the headline `output_tok_s` reflects steady-state decode bandwidth.

4. **Sharegpt tok/s (8,279) > decode tok/s (7,293)** despite sharegpt having shorter avg outputs (~576 vs 4096). At c=512 the engine processes more requests in parallel; the per-step decode tensor is wider. Throughput per GPU hour is what matters, and sharegpt extracts more of it on this rig at this concurrency.

5. **The autopilot took the fallback path correctly** — it identified that `run_matrix.sh` was hardcoded to gpt-oss-20B, wrote a bespoke 3-cell bash loop, used the right tokenizer, and pointed at `~/vllm-tune/data/datasets/sharegpt_prompts.txt` as the only sharegpt dataset on the box. This validates the architecture for arbitrary new models — the load generator and Docker image are generic, only the orchestration glue needs per-model attention. (Note: the autopilot did stall in the writeup phase after the bench completed; that's a separate Claude-driver issue, not an architecture issue.)

## Verdict

**Inconclusive.** The hypothesis required 100% success on all three cells; only 2/3 cleared (decode and sharegpt). The prefill cell failed at the **client harness layer** with `Errno 24: Too many open files`, not at the server. The BASE configuration was never actually stress-tested at prefill c=1024.

This is `inconclusive` rather than `refuted` because:
- vLLM stayed healthy throughout — the boot log shows successful 200 OK responses across all three cells.
- The CSV row for prefill is `success_pct=0.0` with all metrics `-1` because evalscope couldn't even establish connections, not because the engine returned errors.
- A re-run with `ulimit -n 65536` (or with c≤512) would settle the question.

Lossless gate: not yet validated (gap is acknowledged in the hypothesis; lm-eval wiring is slice-10 work).

## Next hypotheses

1. `qwen2.5-7b-base-rerun-on-h100` — re-run BASE with `ulimit -n 65536` before the sweep, OR drop prefill concurrency from c=1024 to c=512. Settles the inconclusive verdict.
2. `qwen2.5-7b-prefix-caching-on-h100` — `--enable-prefix-caching` on top of BASE. Expected gain: throughput improvement on the sharegpt workload (which has agentic-style prefix reuse).
3. `qwen2.5-7b-chunked-prefill-on-h100` — `--enable-chunked-prefill` on top of BASE. Expected: prefill TTFT improvement at high concurrency, possible decode hit.
4. `qwen2.5-7b-fp8-kv-on-h100` — `--kv-cache-dtype fp8` on top of BASE. Expected: 2× KV capacity → higher max concurrency, with negligible accuracy delta.

These four are the natural single-knob ablations from BASE for this model. None of them require an Eagle3 draft (no public Qwen-7B Eagle3 speculator) so they're all viable on this rig today.
