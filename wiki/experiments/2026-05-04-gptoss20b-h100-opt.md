First end-to-end Mac-driver round backed by `vllm-tune`'s `run_matrix.sh` orchestrator. The OPT config booted on vLLM 0.19.0 in 192 s, ran all three per-replica workloads in 6 min wall-clock, and produced complete metric coverage across decode / prefill / sharegpt.

## Hypothesis under test

[`gptoss-20b-opt-baseline`](../hypotheses/gptoss-20b-opt-baseline.md): the OPT config from `vllm-tune`'s 9-config matrix establishes a clean per-replica baseline on a single H100 80 GB SXM at `gpu_memory_utilization=0.85`.

## Setup

**Host:** `h100-1` (`adarsh@89.169.110.238`) — 1× NVIDIA H100 80 GB HBM3, driver 580.126.09, CUDA 13.0.

**Image:** `vllm/vllm-openai:v0.19.0` (digest `sha256:d9a5c1c1614c959fde8d2a4d68449db184572528a6055afdd0caf1e66fb51504`).

**Models on disk** (`/srv/gptoss-models/`): `openai/gpt-oss-20b` (39 GB), `RedHatAI/gpt-oss-20b-speculator.eagle3` (1.6 GB).

**Command** (run from `~/vllm-tune`, `.venv/` providing `evalscope[perf]==1.6.1`):

```bash
DOCKER="sudo docker" \
  bash scripts/experiments/gptoss20b_config_search/run_matrix.sh \
    --vllm-version v0.19.0 \
    --gpu-mem 0.85 \
    --configs OPT
```

**Container launch flags** (verbatim from `boot_logs/OPT.log`):

```
docker run --gpus all --ipc=host --shm-size 16g -p 8000:8000 \
  -v /srv/gptoss-models:/models:ro \
  --entrypoint python3 vllm/vllm-openai:v0.19.0 \
    -m vllm.entrypoints.openai.api_server \
    --model /models/openai/gpt-oss-20b --served-model-name gptoss \
    --host 0.0.0.0 --port 8000 \
    --tensor-parallel-size 1 --trust-remote-code \
    --gpu-memory-utilization 0.85 \
    --dtype bfloat16 --kv-cache-dtype fp8 \
    --block-size 128 --max-num-batched-tokens 16384 \
    --speculative-config '{"method":"eagle3","model":"/models/RedHatAI/gpt-oss-20b-speculator.eagle3","num_speculative_tokens":3}'
```

Engine init took 99.72 s (model load + flashinfer autotuner + 81 PIECEWISE + 81 FULL CUDA graphs captured). API ready at `t=192 s`.

**Diff vs stock vLLM 0.19.0 defaults:** `bf16 + fp8 KV + block_size=128 + max_num_batched_tokens=16384 + Eagle3-RedHatAI k=3 + gpu_memory_utilization=0.85 (vs 0.9 default)`.

## Observations

- **vLLM 0.19.0 + `gpu_memory_utilization=0.85` boots cleanly on 80 GB SXM** for gpt-oss-20B + Eagle3 draft + KV cache. Confirms the v0.19 sampler-warmup-budget fix described in `gptoss-20b-80gb-h100-oom-finding.md` works on driver 580 / CUDA 13.
- **CUDA-graph pool over-estimate**: vLLM logged `CUDA graph pool memory: 1.61 GiB (actual), 1.07 GiB (estimated), difference: 0.53 GiB (33.2%)`. Estimator under-counts by ~⅓; matches the note in `vllm-tune/deploy/vllm-optimized/entrypoint.sh` rationale for the 0.95 → 0.90 retreat under driver 575 / CUDA 12.9.
- **Decode is TTFT-dominated at c=128** (TTFT p50 = 2.37 s, TPOT p50 = 5.5 ms). The first-token wait represents ~9.5% of E2E p50. Throughput-limited workloads on this rig won't benefit much from TTFT optimization at this concurrency.
- **Prefill cell is the bottleneck workload**: 86 ms TPOT p50 (15× decode TPOT) and 20 s TTFT p50. 4 K input × 512 prompts × c=512 saturates prefill compute.
- **Eagle3 k=3 acceptance is roughly 0.77 × 0.625 ≈ 48% effective at draft position 2**, declining with depth. Whether dropping to k=2 helps net throughput is the natural follow-up (the K2 hypothesis in the matrix).

## Verdict

**Supported.** The hypothesis claimed `OPT` would boot and serve all three workloads with 100% success on 1× H100 80 GB at `gpu_memory_utilization=0.85`. All three cells passed. Per-replica numbers are now in the wiki as a baseline future experiments measure against.

## Next hypotheses

- `gptoss-20b-k2-on-h100`: K2 (Eagle3 k=2 instead of k=3). Predicted ≥ +5% sharegpt tok/s, ≤ −2% decode tok/s — the LEAN config in deploy/ already chose k=2 as the workload-agnostic winner; this puts that decision into the wiki audit trail.
- `gptoss-20b-batch8k-on-h100`: BATCH8K (max_num_batched_tokens=8192 instead of 16384). 16 K matmul flop-bound finishes too fast on 20 B; predicted prefill regression but possible decode ~equal.
- `gptoss-20b-nospec-on-h100`: NOSPEC. Isolates whether speculation itself is the source of the prefill TTFT cost we just measured (20 s p50 is steep).
- `gptoss-20b-lean-on-h100`: LEAN — the current shipping default (k=2 + blk=64 + batch=8k + bf16 + fp8 KV). If this beats OPT cleanly the wiki re-confirms what `vllm-tune/deploy/vllm-optimized/` already encodes.
