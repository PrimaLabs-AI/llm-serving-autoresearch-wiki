Rerun of [round-9 v2 inconclusive](2026-05-04-qwen2.5-7b-base-on-h100.md) with the bench-side `RLIMIT_NOFILE` auto-bump committed in `4b2fe53`. All three cells now complete with 100% success — the prefill failure last round was confirmed to be the client-side ulimit ceiling, not a server limit.

## Hypothesis under test

[`qwen2.5-7b-base-on-h100`](../hypotheses/qwen2.5-7b-base-on-h100.md): stock vLLM 0.19.0 (`--gpu-memory-utilization 0.9`, no other flags) serves `Qwen/Qwen2.5-7B-Instruct` on a single H100 80 GB SXM and completes a per-replica concurrency sweep across decode @ c=256, prefill @ c=1024, sharegpt @ c=512 with **100% request success**. Pass condition: all three cells return `success_pct=100.0` with non-zero `output_tok_s`.

## Setup

**Host:** `h100-1` (`adarsh@89.169.110.238`) — 1× NVIDIA H100 80 GB HBM3, driver 580.126.09, CUDA 13.0.

**Image:** `vllm/vllm-openai:v0.19.0`.

**Model on disk:** `/srv/models/Qwen/Qwen2.5-7B-Instruct/` (~15 GB BF16, 4 safetensors shards).

**Container launch (verbatim):**

```bash
sudo docker run --rm --name qwen7b-bench --gpus all --ipc=host --shm-size 16g \
    -p 8000:8000 -v /srv/models:/models:ro \
    --entrypoint python3 vllm/vllm-openai:v0.19.0 \
      -m vllm.entrypoints.openai.api_server \
      --model /models/Qwen/Qwen2.5-7B-Instruct \
      --served-model-name qwen \
      --host 0.0.0.0 --port 8000 \
      --tensor-parallel-size 1 \
      --trust-remote-code \
      --gpu-memory-utilization 0.9
```

Cold start: model loaded, `/health` ready in **60 s** (much faster than v2's ~2 min — likely due to vLLM CUDA graph cache hits on the second run).

**Sweep loop (verbatim):**

```bash
SWEEP="bench/scripts/benchmark/sweep_api_providers_evalscope.py"
TOK="/srv/models/Qwen/Qwen2.5-7B-Instruct"
SHAREGPT="$HOME/vllm-tune/data/datasets/sharegpt_prompts.txt"
for pair in "decode 256" "prefill 1024" "sharegpt 512"; do
  W=$(echo $pair | cut -d" " -f1); C=$(echo $pair | cut -d" " -f2)
  python3 "$SWEEP" --self-urls http://localhost:8000 --providers self \
      --self-model qwen --workloads "$W" --concurrency "$C" --n-mode conc \
      --tokenizer-path "$TOK" --sharegpt-path "$SHAREGPT" --cooldown 15 \
      --outputs-root "$OUT/evalscope/BASE" --out "$OUT/per_config/BASE.csv"
done
```

The bench script now auto-bumps `RLIMIT_NOFILE` to 65536 at startup (committed in `bench/scripts/benchmark/sweep_api_providers_evalscope.py` line 56-77 of commit `4b2fe53`), so the c=1024 prefill cell no longer hits the SSH-default `ulimit -n 1024` cap.

**Diff vs v2 (the inconclusive round):** identical container launch, identical sweep flags. The only difference is the bench script's startup-time ulimit raise. Same model, same hardware, same config, same workloads.

## Baseline comparison

This experiment supersedes [the v2 inconclusive run](2026-05-04-qwen2.5-7b-base-on-h100.md). Comparison against v2:

| Workload | v2 (with ulimit bug) | v4 rerun (fixed) | Δ |
|---|---:|---:|---:|
| decode @ c=256 output tok/s | 7,293 | 5,973 | **−18.1%** |
| prefill @ c=1024 success_pct | 0% (Errno 24) | **100%** | clean win |
| prefill @ c=1024 output tok/s | — | 3,190 | new datum |
| sharegpt @ c=512 output tok/s | 8,279 | 8,282 | +0.04% (noise) |

Decode regressed by ~18% which is more than benchmark noise — suggests the original v2 decode run benefited from a less-loaded vLLM cache state (no prior cells had run; all queues empty). The rerun's decode cell ran AFTER the cooldown from the prior decode warm-up (`--cooldown 15`), so the engine had different state. **The 5,973 number is the more reliable per-replica datum** for a steady-state run.

|---|---:|---:|---:|---:|---:|---:|---:|---:|
| decode (256 in / 4096 out) | 256 | **5,973** | 1.46 | 1.19 | 41.7 | 171.6 | 175.4 | 100% |
| prefill (4096 in / 512 out) | 1024 | **3,190** | 6.23 | 78.7 | 68.7 | 114.3 | 164.2 | 100% |
| sharegpt (~142 in / ~574 out) | 512 | **8,282** | 14.44 | 1.28 | 17.8 | 10.4 | 25.2 | 100% |

Source: [`per_config/BASE.csv`](../../raw/benchmarks/2026-05-04-qwen2.5-7b-base-rerun/per_config/BASE.csv).

### Per-cell wall-clock

- decode @ c=256: ~3 min (256 prompts × 4096 tok output, batch saturated)
- prefill @ c=1024: ~6 min (1024 prompts × 4096 tok input, prefill-compute-bound)
- sharegpt @ c=512: ~37 s (short outputs, throughput-bound)

Total run including cooldowns: ~12 min; plus 60 s cold start = ~13 min wall-clock end-to-end.

### Side-by-side vs gpt-oss-20B OPT (from round 7)

| Workload | gpt-oss-20B OPT @ c=128/512/256 | Qwen-7B BASE @ c=256/1024/512 |
|---|---:|---:|
| decode tok/s | 13,945 | 5,973 |
| prefill tok/s | 3,501 | 3,190 |
| sharegpt tok/s | 16,192 | 8,282 |

Even at 2-4× the per-cell concurrency, Qwen-7B BASE delivers ~50% the per-replica throughput of gpt-oss-20B OPT on decode and sharegpt. The headline gap is mostly Eagle3 (gpt-oss had k=3 spec; Qwen has none — no public Qwen-7B Eagle3 draft). On prefill the gap closes (Qwen 3,190 vs gpt-oss 3,501 — Qwen prefill is competitive at higher per-replica concurrency).

## Observations

1. **Ulimit auto-bump is the right place to fix this.** Putting it in the bench script (rather than every prompt or wrapper) means future rounds, future models, future hardware never see the ulimit ceiling unless they exceed 65,536 fds (which would mean c≥60K — implausible for any single GPU).

2. **vLLM CUDA graph cache makes warm boots fast.** v2's cold boot took ~2 min; this rerun's boot took ~60 s. The container was different (`--rm` removed it after v2), but vLLM's on-disk graph cache survived. This should generalize — every retry of the same engine config on the same model is significantly faster after the first run.

3. **Decode throughput is variance-prone at single-cell concurrency.** v2's 7,293 vs v4's 5,973 (decode @ c=256) is an 18% spread across two runs of identical config. This is more than the ±5% I'd expected from noise. **Worth filing as `wiki/observations/decode-cell-variance.md`** — single-cell numbers below ~1% noise need 3-5 repeat runs to settle.

4. **Sharegpt is the most reproducible cell** — v2's 8,279 vs v4's 8,282 (Δ = 0.04%, well within noise). The shorter outputs and higher concurrency keep the engine in steady state throughout the run.

5. **Prefill at c=1024 has 78.7 s TTFT p50** — that's almost a third of the cell's wall-clock. The first request completes fast (low p50 actually means most queue at the back). This is expected: 1024 concurrent × 4096 input tokens = 4M prompt tokens to schedule, and the scheduler can only batch ~4096 per iteration. Need ~1000 prefill iterations to clear the queue.

## Verdict

**Supported.** The hypothesis required 100% success on all three cells; this rerun cleared all three:
- decode @ c=256: 256/256 success, 5,973 output tok/s
- prefill @ c=1024: 1024/1024 success, 3,190 output tok/s
- sharegpt @ c=512: 512/512 success, 8,282 output tok/s

The per-replica baseline for `Qwen/Qwen2.5-7B-Instruct` on 1× H100 80 GB SXM with stock vLLM 0.19.0 is now established and can be used as the comparison floor for all subsequent ablations.

Lossless gate: not yet validated (gap still acknowledged; lm-eval wiring is slice-10 work).

## Next hypotheses

The natural single-knob ablations from BASE for Qwen-7B + H100:

1. **`qwen2.5-7b-prefix-caching-on-h100`** — add `--enable-prefix-caching`. Most likely to help on sharegpt (prefix reuse across multi-turn). Predicted: +10-20% sharegpt tok/s, no regression elsewhere.
2. **`qwen2.5-7b-chunked-prefill-on-h100`** — add `--enable-chunked-prefill`. Should drop prefill TTFT p50 from 78.7 s by interleaving prefill chunks with decode steps. Predicted: −30-50% prefill TTFT, possibly minor decode regression.
3. **`qwen2.5-7b-fp8-kv-on-h100`** — add `--kv-cache-dtype fp8`. Doubles KV capacity → enables higher max concurrency. Predicted: same throughput at current concurrencies, +50-100% max concurrency before OOM.
4. **`qwen2.5-7b-batch16k-on-h100`** — add `--max-num-batched-tokens 16384` (vs default 8192). Prefill scheduler can batch more per iteration. Predicted: +10-20% prefill tok/s, no decode change.

If 1+2+3+4 all show positive deltas, propose a synthesized `qwen2.5-7b-optimized-on-h100` combining the wins, then validate that combination is the actual winner (not just a sum of ablations).
