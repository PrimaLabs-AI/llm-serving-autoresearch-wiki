First end-to-end run of the Gemma 4 E4B torchax fine-tune trainer on a TPU v6e-**4** host (not v6e-8 as the program scaffold assumed). Establishes the performance floor the rest of the program optimizes against.

## Hypothesis under test

"A torchax + JAX trainer built from the jax-huggingface Part 2 recipe can fine-tune Gemma 4 E4B on wikitext under FSDP, capture an xprof profile, and produce an interpretable step-time number." — i.e., **infrastructure check**, not a perf hypothesis.

## Setup

- Hardware: 4 × TPU v6e (measured; not the program-README default of 8).
- Conda env: `gemma4_py313` (Python 3.13.13).
- Dependencies: jax 0.10.0 + libtpu 0.0.40; torch 2.11.0+cpu; torchax 0.0.12 (editable install from wiki submodule commit `8f957d1`); transformers 5.7.0.dev0 (from main); datasets 4.8.4; optax 0.2.8; accelerate 1.13.0.
- Strategy: `fsdp` (1D mesh, fsdp=4). The **default was flipped from TP=8 to FSDP** in this run — see "Sharding strategy change" below.
- Dataset: `wikitext-2-raw-v1`, packed at the given seq_len.
- XLA flags: none beyond defaults. `LIBTPU_INIT_ARGS` unset. **No XLA optimizations applied.**
- Model load path: `Gemma4ForConditionalGeneration` (the HF repo declares `architectures: ['Gemma4ForConditionalGeneration']`); forward is monkey-patched to a text-only path (`model.language_model` + `lm_head` + `final_logit_softcapping=30.0`) to bypass the multimodal mask.
- Command:

  ```bash
  python wiki/experiments/gemma4_autoresearch_optimization/torchax/train.py \
    --steps 10 --batch_size 1 --seq_len <S> --strategy fsdp
  ```

## Results

Seq-length sweep at per-chip `batch=1` (global batch = 4):

| seq | tokens/step (global) | step 0 compile | step 1 recompile | steady-state step | tokens/sec | est. FLOPs/step | **MFU** | loss trajectory |
|---|---|---|---|---|---|---|---|---|
| 512  | 2,048 | 112 s | 113 s | **117 ms**  | 17,500 | ~57 TFLOPs  | **~13%** | 9.33 → 4.91 (training) |
| 1024 | 4,096 | 148 s | 151 s | **134 ms**  | 30,570 | ~115 TFLOPs | **~23%** | 3.93 → 2.04 (training) |
| 2048 | 8,192 | 156 s | 158 s | **249 ms**  | 32,900 | ~240 TFLOPs | **~26%** | **NaN** (see below) |

FLOPs counted by detailed matmul enumeration (per-layer × 42 layers + LM head) × 3 for `2× backward + 1× forward`. The naïve `6PT` approximation with `P=8B` overestimates by ~1.6× because Gemma 4's headline "~8B" includes the Per-Layer-Embedding (PLE) lookup tables which do not participate in matmul FLOPs. The true matmul-`P` is closer to 4B.

### Peak-FLOP reference

v6e peak bf16 (MXU): ~918 TFLOP/s per chip → 3,672 TFLOP/s aggregate across 4 chips. MFU = `achieved / peak`.

### Compile-time observations

- Step 0 compiles the full train graph (156 s at seq=2048).
- **Step 1 *re*compiles** — takes ~157 s again. Likely a sharding-spec mismatch between step-0's donated inputs and step-1's (un-donated) inputs, or a second-trace triggered by an optimizer-state-shape change. Steps 2+ hit the cache and run at the steady-state number. Dropping this redundant compile is low-effort — candidate follow-up.

## Sharding strategy change

The scaffold originally defaulted to **TP=8** (Megatron tensor parallel from jax-hf Part 2). Changed to **FSDP** (default) for this baseline — FSDP is the standard for fine-tuning: params sharded across the `fsdp` axis, gathered on-demand per layer during forward, reduce-scattered during backward. The TP path remains available via `--strategy tp`.

For Gemma 4 E4B (~8B params in bf16 = 16 GB) on v6e-4 (128 GB aggregate HBM): FSDP keeps weights at ~4 GB/chip and optimizer state (AdamW fp32 ≈ 16 bytes/param × 8B = 128 GB aggregate) at ~32 GB/chip — tight but feasible. TP-only would replicate optimizer state and OOM.

**Sharding plan bucket counts** (FSDP, `fsdp=4`):

| Bucket | Count |
|---|---|
| `fsdp_shard` (largest dim divisible by 4) | 1,107 |
| `replicated` (1D tensors, norms) | 928 |
| `undivisible` (mostly 42 × `layer_scalar[1]`) | 42 |

## Observations

1. **NaN loss at seq≥2048.** Loss is non-NaN and decreasing at seq=512 and seq=1024; it is NaN from step 0 at seq=2048 — forward produces NaN before any gradient update. Applying `final_logit_softcapping=30.0` in the monkey-patched forward did **not** fix it. Likely candidates: bf16 attention-score overflow at longer sequences, or a masking/position-embedding edge case in Gemma 4's hybrid local-SW-512 + global attention at seq>SW. Recorded as the first hypothesis this baseline exposes.
2. **OOM at batch=4, seq=2048.** Global batch 16 hits HBM limit during trace — attention matrix is materialized (no flash/splash). Motivates **hypothesis #1** (tokamax Splash Attention / `jax.experimental.pallas.ops.tpu.splash_attention`) to unlock larger batch.
3. **MFU scaling with seq length** (13% → 23% → 26%). FSDP collective overhead amortizes over more compute at longer sequences; short sequences are communication-dominated.
4. **Compile time dominates for ≤ ~10-step runs.** Two ~150-second compiles (300 s overhead vs 2.5 s of useful work at seq=2048). Motivates **hypothesis #7** (scan-over-layers) to collapse the per-layer unrolled compile into one.
5. **The HF checkpoint is multimodal-only.** `Gemma4ForCausalLM.from_pretrained(google/gemma-4-E4B)` silently re-initializes ~all weights (key-name mismatch against the `model.language_model.*` prefix in the checkpoint). Detected by loss starting at ~log(vocab)=12.5. Solution: load `Gemma4ForConditionalGeneration` (weights match) and bypass the multimodal orchestrator via monkey-patched forward.

## Verdict

**Supported** for the infrastructure-check hypothesis: trainer runs, produces metrics, captures profile. **Not a valid baseline for quality metrics** at seq=2048 until NaN is fixed.

## Next hypotheses generated

1. **Fix NaN loss at seq≥2048** (precondition for any perf work at seq=2048).
   - Try fp32 loss computation (cast logits to fp32 before softmax) — should be a 1-line change.
   - Try eager attention vs SDPA impls.
   - Compare per-layer hidden-state stats at seq=1024 vs seq=2048 to localize where NaN begins.
2. **Drop the step-1 recompile.** Pre-specify `in_shardings` / `out_shardings` on `jax.jit` to match step-0's output sharding so step 1 hits the cache.
3. Original program-level hypotheses remain open — see [`README.md`](../../README.md) for the ranked list. Hypothesis #1 (Splash Attention) and #7 (scan-over-layers) are the highest-leverage given what this baseline revealed.

## Scaffold changes applied during this run

Tracked so future runs use the same code. All changes in `wiki/experiments/gemma4_autoresearch_optimization/torchax/`:

| Change | File | Reason |
|---|---|---|
| Default strategy flipped TP=8 → FSDP | `train.py`, `model/sharding.py` | FSDP is the correct fine-tune default; TP=8 assumes v6e-8 and caps optimizer-state replication. |
| FSDP sharding rules added | `model/sharding.py` | Shard every ≥2D param on largest divisible dim over `'fsdp'`. |
| Torchax `_jax_view` → `jax_view` | `train.py` (3 sites) | Pytree-map version; the single-value variant left torch tensors unconverted, breaking optax. |
| Drop `device_map="cpu"` | `train.py` | Not needed, and required `accelerate` install. |
| Prefer `Gemma4ForConditionalGeneration` | `train.py` | The HF checkpoint is multimodal-only; loading `Gemma4ForCausalLM` silently re-inits. |
| Text-only forward monkey-patch | `train.py` | Bypass the multimodal orchestrator (boolean `input_ids[mask] = pad` is not JIT-able). |
| `final_logit_softcapping=30.0` in forward | `train.py` | Gemma 4's own forward applies it; our text-only path must too. (Did not fix the seq=2048 NaN by itself.) |
| `jax.sharding.use_mesh(mesh)` context | `train.py` | Make the mesh active for `jax.jit` / `device_put`. |
| `torchax` install from local submodule | `requirements.txt` | Original pointed at `pytorch/xla` subpath; repo is `google/torchax`. |
