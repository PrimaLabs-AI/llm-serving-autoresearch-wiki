Round 1 of the Qwen-7B 3-round autopilot. `--enable-prefix-caching` ran cleanly on all three workloads, but the realized prefix cache hit rate was **0.78%** (40K hits / 5.2M queries), so the predicted +10-20% sharegpt throughput improvement didn't materialize — sharegpt @ c=512 came in at +0.9% vs BASE, well within run-to-run noise. **The hypothesis is refuted on the test workloads, but the underlying effect (prefix caching saves recompute when prefixes are reused) is not disproven** — our test inputs simply don't have prefix overlap.

## Hypothesis under test

[`qwen2.5-7b-prefix-caching-on-h100`](../hypotheses/qwen2.5-7b-prefix-caching-on-h100.md): adding `--enable-prefix-caching` delivers ≥10% improvement on sharegpt `output_tok_s` at c=512 vs BASE (8,282 tok/s), with decode/prefill within ±5% of BASE.

Pass: sharegpt output_tok_s ≥ 9,110 (1.10×) AND decode + prefill within 95-105% of BASE.

## Setup

**Host:** `h100-1` — 1× NVIDIA H100 80 GB HBM3.
**Image:** `vllm/vllm-openai:v0.19.0`.
**Model on disk:** `/srv/models/Qwen/Qwen2.5-7B-Instruct/`.

**Container launch (verbatim):**

```bash
sudo docker run -d --name qwen7b-pfxcache --gpus all --ipc=host --shm-size 16g \
    -p 8000:8000 -v /srv/models:/models:ro \
    --entrypoint python3 vllm/vllm-openai:v0.19.0 \
      -m vllm.entrypoints.openai.api_server \
      --model /models/Qwen/Qwen2.5-7B-Instruct \
      --served-model-name qwen \
      --host 0.0.0.0 --port 8000 \
      --tensor-parallel-size 1 \
      --trust-remote-code \
      --gpu-memory-utilization 0.9 \
      --enable-prefix-caching
```

**Diff vs BASE:** added `--enable-prefix-caching`. Everything else identical.

## Baseline comparison

### Prefix cache hit rate (the load-bearing number)

From [`metrics/PFXCACHE.txt`](../../raw/benchmarks/2026-05-04-qwen2.5-7b-prefix-caching-on-h100/metrics/PFXCACHE.txt) at end-of-run:

| Counter | Value |
|---|---:|
| `vllm:prefix_cache_queries_total` | 5,205,286 |
| `vllm:prefix_cache_hits_total` | 40,576 |
| **Hit rate** | **0.78%** |

### What that means

Prefix cache improves throughput by `(hit_rate × prefill_compute_savings)`. With a 0.78% hit rate, the upper-bound improvement is also ~0.78%. The +0.9% measured on sharegpt is consistent with this; the +0.2% on prefill is consistent with this. The +34.1% decode "improvement" is run-to-run variance — BASE decode itself ranged from 5,973 (rerun) to 7,293 (v2 inconclusive) at the same config, a ~22% spread.

The dataset shapes generated by `evalscope --dataset random` (decode, prefill) and `--dataset sharegpt` (sampled from `sharegpt_prompts.txt`) don't have meaningful prompt-prefix overlap. The workloads where prefix caching shines (multi-turn agentic with shared system prompts, code completions sharing repository context, RAG with repeated retrieved chunks) aren't represented in this sweep.

## Observations

1. **Prefix cache hit rate is the definitive feedback signal** — measure it directly via `/metrics` rather than inferring from throughput delta. 0.78% hit rate explains the ~1% throughput delta with no further analysis needed.

2. **Run-to-run variance dominates the decode signal at single-cell concurrency.** Three identically-configured BASE-shape runs gave decode @ c=256 of 7,293 / 5,973 / 8,009 = ±18% spread. Differential studies on this workload at this concurrency need 3+ repeats per config to detect <20% effects. **File `wiki/observations/decode-cell-variance-h100.md`** as a permanent prior.

3. **Prefix caching doesn't regress the workloads where it has nothing to do** — TPOT, TTFT, success rate all within noise vs BASE. The book-keeping overhead is genuinely small.

4. **The hypothesis was about the wrong workload.** Prefix caching is a real, useful feature, but the evalscope `random` and shuffled-sharegpt datasets aren't representative of the actual production patterns where it matters (multi-turn chat, agent loops with growing context, code completion sharing project context). To test it properly we'd need either: (a) a synthetic workload with known shared prefixes, (b) a real multi-turn workload (chat history → response → new query → response, replayed), or (c) a longer-tail conversation dataset where users send the same system prompt many times.

## Verdict

**Refuted (on these workloads).** Pass condition required sharegpt ≥ 9,110 tok/s; got 8,357 (+0.9%). Hypothesis as stated is refuted. Note: this does NOT refute the broader claim that prefix caching helps — the measurement was constrained by the test workload not having prefix overlap (0.78% hit rate). A retest with a workload designed for prefix reuse would settle the broader question.
