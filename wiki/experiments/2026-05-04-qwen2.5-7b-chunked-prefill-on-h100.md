Round 3 of the Qwen-7B 3-round autopilot. Test ran cleanly on all 3 cells, but the boot log reveals that **`--enable-chunked-prefill` is a no-op on vLLM 0.19.0** — chunked-prefill is enabled by default. The "BASE" rerun was already running with chunked-prefill on, so this experiment compared identical configurations and (correctly) measured no difference. Verdict: **invalid** (the knob under test wasn't actually toggled). Forces reinterpretation of round 1 (prefix-caching) too — same default-on issue.

## Hypothesis under test

[`qwen2.5-7b-chunked-prefill-on-h100`](../hypotheses/qwen2.5-7b-chunked-prefill-on-h100.md): adding `--enable-chunked-prefill` to BASE delivers ≥30% reduction in prefill TTFT p50 at c=1024 vs BASE (78.7 s).

The hypothesis explicitly flagged this risk: *"vLLM 0.19.0 may already enable chunked prefill by default for some configs. Verify in the boot log: `Chunked prefill is enabled` should appear if active. If BASE was already running with it, this hypothesis is a no-op and should be marked `inconclusive`."* That risk fired.

## Setup

Identical to BASE rerun except added `--enable-chunked-prefill`. Both boot logs grep'd:

```bash
$ grep -E "chunk|prefix" boot_logs/CHUNKED.log
INFO 05-04 13:41:32 [utils.py:233] non-default args: {'enable_chunked_prefill': True}
INFO 05-04 13:41:40 [scheduler.py:238] Chunked prefill is enabled with max_num_batched_tokens=8192.
… enable_prefix_caching=True, enable_chunked_prefill=True …

$ grep -E "chunk|prefix" ../2026-05-04-qwen2.5-7b-base-rerun/boot_logs/BASE.log
INFO 05-04 12:09:01 [scheduler.py:238] Chunked prefill is enabled with max_num_batched_tokens=8192.
… enable_prefix_caching=True, enable_chunked_prefill=True …
```

**Both runs have `enable_chunked_prefill=True` and `enable_prefix_caching=True`** (the latter despite BASE not passing `--enable-prefix-caching`). vLLM 0.19.0 defaults both flags on. The CHUNKED run's boot log notes "non-default args" only mentions `enable_chunked_prefill: True` — meaning it was set explicitly, but that's a redundant set; it was already on by default.

## Baseline comparison

Baseline = [Qwen2.5-7B BASE rerun](2026-05-04-qwen2.5-7b-base-rerun.md). Same hardware, same image.

## Observations

**This is a methodology finding, not a knob finding.**

1. **vLLM 0.19.0 defaults `enable_chunked_prefill=True` and `enable_prefix_caching=True`.** Confirmed in two independent boot logs (BASE rerun + CHUNKED). The `--enable-X` flags merely re-affirm what the engine is already doing. To actually test the impact of these features, we'd need explicit DISABLE flags (e.g., `--no-enable-chunked-prefill`) and compare default-on vs default-off.

2. **This invalidates round 1's PFXCACHE verdict.** Round 1 measured prefix-caching's effect by adding `--enable-prefix-caching` and got ~0% improvement. We marked it `refuted`, but the actual reason is the same: the flag is a no-op because prefix-caching was already on in BASE. The 0.78% prefix-cache hit rate isn't because the workload has no overlap (though it doesn't); it's because **the default-on prefix-cache hit rate IS 0.78% on this workload** — a ceiling, not the absence of the feature. Need to file a hypothesis follow-up: `qwen2.5-7b-disable-prefix-caching-on-h100` to measure the actual delta.

3. **The "BASE" config is misnamed.** Stock vLLM 0.19.0 with no flags ≠ stock minus optimizations. vLLM has aggressively defaulted recent serving optimizations on. Future BASE-comparison hypotheses must check the engine config in the boot log to know what's actually under test.

4. **Decode cell variance is genuinely massive at this single-cell c=256 size.** Four BASE-shape runs gave 5,973 / 7,293 / 8,009 / 7,258 = ±18% spread. Anything claimed in the ±20% band is meaningless without 3-5 repeat measurements.

## Verdict

**Invalid.** The hypothesis was about toggling chunked-prefill on/off; both runs had it on. The test didn't measure what we intended. The prefill TTFT figure (78.5 s vs 78.7 s) is the right number for the *current default vLLM 0.19 config* but does NOT establish how chunked-prefill itself contributes — that requires a future test with `--no-enable-chunked-prefill`.

## Next hypotheses

1. **`qwen2.5-7b-disable-chunked-prefill-on-h100`** — explicitly disable chunked-prefill with `--enable-chunked-prefill=false` (or whatever vLLM's negation is) and measure how much *worse* prefill TTFT and decode become. This is the proper "what does chunked-prefill actually do" test.
2. **`qwen2.5-7b-disable-prefix-caching-on-h100`** — same logic for prefix-caching. Could be more impactful since the cache currently uses some KV memory even at low hit rates.
3. **`qwen2.5-7b-fp8-kv-on-h100`** (already queued, rank 1 after this commits) — fp8 KV is **off** by default, so this is a real test of an untoggled knob. Predicted: ≤5% throughput change at current concurrencies, +50-100% concurrency headroom.
4. **`qwen2.5-7b-batch16k-on-h100`** — `--max-num-batched-tokens 16384` (vs default 8192). The boot log confirms 8192 is the current default. Predicted: +10-20% prefill tok/s.

The fp8 KV hypothesis (rank 1) is the right next round — it's the first **untoggled** knob we have queued.
