After three autopilot ablation rounds (prefix-caching, prefix-caching repeat, chunked-prefill) on Qwen2.5-7B + 1× H100, the headline finding is **methodological**, not numerical: vLLM 0.19.0 has aggressively enabled both `enable_prefix_caching` and `enable_chunked_prefill` by default, so two of the three single-knob ablations we queued were no-op tests. The optimization frontier we measured is the same as the frontier vLLM ships out of the box. To make further progress we need different ablations.

## The single load-bearing finding

vLLM 0.19.0's serving config defaults (in the boot log of every Qwen-7B run we did):

```
enable_prefix_caching=True
enable_chunked_prefill=True
max_num_batched_tokens=8192
kv_cache_dtype=auto       # = bf16 for the model dtype, NOT fp8
```

Three of these have been promoted to defaults since the gpt-oss-20B v019 study was conducted ([source](../sources/2026-04-gptoss-20b-v019-findings.md), where they were OPTimized features). The fourth (`kv_cache_dtype`) remains opt-in.

This means our two next-round picks (prefix-caching, chunked-prefill) wanted to test features that no longer require explicit opt-in. The actual remaining configuration leverage on Qwen-7B + vLLM 0.19 + H100 80 GB is much narrower than the seed hypotheses assumed.

## What's actually testable

1. **fp8 KV cache** (rank 1 of remaining queue, [hypothesis](../hypotheses/qwen2.5-7b-fp8-kv-on-h100.md)) — `--kv-cache-dtype fp8` is **off** by default. This is the next genuine ablation. Expected: ≤5% throughput change at current concurrencies, doubled KV cache capacity → +50-100% max concurrency before OOM.

2. **`max_num_batched_tokens` larger** (e.g., 16384) — boot log confirms 8192 is current default. Predicted: +10-20% prefill tok/s by feeding more tokens per scheduler iteration. New hypothesis [`qwen2.5-7b-batch16k-on-h100`](#) to file.

3. **Explicit DISABLE tests** — `--enable-prefix-caching=false` and `--enable-chunked-prefill=false` to measure how much these now-default features actually contribute. Useful for explaining the *current* BASE numbers, less useful as ablations going forward (since shipping them off would be a regression). New hypotheses to file: [`qwen2.5-7b-disable-prefix-caching-on-h100`](#), [`qwen2.5-7b-disable-chunked-prefill-on-h100`](#).

4. **Higher concurrency cells** — current sweep tops at c=1024 (prefill). The fp8 KV test should unlock c=2048 or c=4096 cells where the headroom advantage actually matters.

5. **Different attention backend** — vLLM defaults are FA2 on H100 (the boot log doesn't show the choice but it's compiled-in for head_dim=128). FA3 is faster but less stable. New hypothesis [`qwen2.5-7b-flashinfer-attention-on-h100`](#) to file.

## What about variance

Across four BASE-shape runs (BASE v2, BASE rerun, PFXCACHE, CHUNKED — all running essentially the same config given the default-on flags), the decode @ c=256 numbers were:

| Run | Output tok/s |
|---|---:|
| BASE v2 (inconclusive) | 7,293 |
| BASE rerun (supported) | 5,973 |
| PFXCACHE (refuted) | 8,009 |
| CHUNKED (invalid) | 7,258 |
| **mean** | **7,133** |
| **stdev** | **848 (~12% RSD)** |

Sharegpt @ c=512 across the same four runs:

| Run | Output tok/s |
|---|---:|
| BASE v2 | 8,279 |
| BASE rerun | 8,282 |
| PFXCACHE | 8,357 |
| CHUNKED | 8,332 |
| **mean** | **8,313** |
| **stdev** | **38 (~0.5% RSD)** |

**Sharegpt is the reliable cell** for differential measurements. **Decode is essentially uninformative below ~20% effect size with a single run** — would need 3-5 repeats per config to detect a real 10% improvement. File [`wiki/observations/decode-cell-variance-h100.md`](#) (pending) as a permanent prior.

## Recommendation

**Stop the current 3-round autopilot here. The remaining data points (fp8 KV, the disable-X tests) are different in shape — they need a fresh autopilot run with the new hypothesis ranking and clean state. Two specific actions:**

1. **Land the methodology fixes first** as a slice 9.6 task: PICK should `git pull` (or read commit-by-commit) before each round to avoid the round-2-style race; observation page on decode variance; new hypothesis pages for the disable-X tests.

2. **Rerun with fp8 KV at rank 1** as a clean 1-round autopilot. The fp8 KV test is well-defined, predicts a clean signal (concurrency headroom is observable directly via `vllm:kv_cache_usage_perc`), and isn't at risk of being a no-op.
